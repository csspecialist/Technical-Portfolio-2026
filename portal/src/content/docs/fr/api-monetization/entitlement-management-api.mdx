## Qu’est-ce que la gestion des droits&nbsp;?

La gestion des droits définit **ce qu’un utilisateur est autorisé à faire** dans une API ou un produit SaaS selon son plan, son abonnement ou ses permissions.

Elle agit comme une **couche de décision entre l’authentification et l’exécution**.

* L’authentification répond à : qui est l’utilisateur&nbsp;?
* Les droits répondent à : que peut-il faire&nbsp;?

Pour comprendre l’authentification, consultez le [Démarrage rapide Auth API](/fr/guides/auth-api-quickstart).

---

## Pourquoi la gestion des droits est essentielle

Sans gestion des droits, il est impossible de contrôler l’accès ou de monétiser une API.

Elle permet de :

* contrôler l’accès aux endpoints et fonctionnalités
* appliquer les plans tarifaires
* soutenir les modèles de monétisation
* prévenir les usages non autorisés

Elle est un composant clé d’un [système de monétisation API](/fr/api-monetization/api-monetization-guide).

---

## Position dans l’architecture

La gestion des droits intervient entre l’authentification et le suivi d’usage.

Flux typique :

1. L’utilisateur s’authentifie
   → [Démarrage rapide Auth API](/fr/guides/auth-api-quickstart)

2. Le système récupère ses droits

3. L’API valide l’accès

4. La requête est acceptée ou refusée

5. Si acceptée, l’usage est enregistré
   → [Architecture de suivi d’usage](/fr/api-monetization/usage-tracking-architecture)

## Implémentation étape par étape

### Étape 1 : Définir les plans

* Gratuit → accès limité
* Pro → fonctionnalités avancées
* Entreprise → accès complet

Voir le [Guide de monétisation API](/fr/api-monetization/api-monetization-guide)

---

### Étape 2 : Modéliser les droits

Chaque utilisateur doit avoir :

* type de plan
* endpoints autorisés
* fonctionnalités activées
* limites d’usage

---

### Étape 3 : Stocker les droits

Options courantes :

* base de données
* cache (Redis)

---

### Étape 4 : Vérifier les droits

```http
GET /api/v1/premium-data
Authorization: Bearer API_KEY
```

Vérifier :

* accès autorisé
* fonctionnalité active
* limite non dépassée

---

### Étape 5 : Gérer les refus

```json
{
  "error": "access_denied",
  "message": "Mettez à niveau votre plan pour accéder à cette ressource."
}
```

---

## Bonnes pratiques

* centraliser la logique
* utiliser un cache
* séparer authentification et autorisation
* concevoir des plans flexibles
* journaliser les vérifications

---

## Erreurs fréquentes

* permissions codées en dur
* confusion auth / autorisation
* absence de mise à jour dynamique
* erreurs peu claires

---

## FAQ

### Qu’est-ce qu’un droit (entitlement) dans une API&nbsp;?

Un droit définit ce qu’un utilisateur peut faire, comme accéder à des endpoints, fonctionnalités ou limites d’usage.

---

### Quelle est la différence entre authentification et droits&nbsp;?

L’authentification vérifie l’identité, les droits définissent les permissions.

---

### Les droits peuvent-ils changer dynamiquement&nbsp;?

Oui, lors d’un changement de plan ou d’abonnement.

---

### Que se passe-t-il si un utilisateur dépasse ses droits&nbsp;?

L’API peut bloquer la requête, retourner une erreur ou déclencher une action de facturation.
